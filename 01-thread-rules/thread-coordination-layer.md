# Thread Coordination Layer 线程协作运行层

本文定义 `CodeWinter` 的线程协作运行层。

它的目标不是替代长期知识层，而是补上多线程运行时的可见性、协作意图表达能力，以及最小必要的反馈回路。

## 1. 目标
线程协作运行层主要解决以下问题：
1. 当前有哪些线程存在
2. 每个线程当前在做什么
3. 哪些线程已阻塞
4. 哪些线程需要协作
5. 当前有哪些协作请求待处理
6. 哪些线程已完成但仍待收口或待归档
7. 哪些线程正在低信心推进或可能偏航

## 2. 设计原则
1. 运行态信息与长期知识层分离
2. 事件驱动更新，而不是流水账更新
3. 每个线程优先只写自己的文件，避免共享写冲突
4. 管理线程负责汇总，不要求执行线程维护全局总览
5. 正式长期知识仍保留人工确认边界
6. 运行态工件不仅要表达“状态”，也要表达“判断质量与下一步建议”

## 3. 组成结构
线程协作运行层放在 `00-control-plane` 下，建议结构如下：
1. `./CodeWinter/00-control-plane/thread-board.md`
   - 管理线程维护的全局线程总览
2. `./CodeWinter/00-control-plane/threads/<thread-id>.md`
   - 每个线程自己的状态卡
3. `./CodeWinter/00-control-plane/collab-requests/<request-id>.md`
   - 每个协作请求自己的请求卡

## 4. 线程状态机
当前推荐固定使用以下状态：
1. `BOOTSTRAPPING`
2. `ACTIVE`
3. `BLOCKED`
4. `NEEDS_COLLAB`
5. `WAITING`
6. `HANDOFF_READY`
7. `DONE`
8. `DORMANT`

说明：
1. `BOOTSTRAPPING`
   - 线程刚接手任务，正在建立上下文
2. `ACTIVE`
   - 线程正在推进当前任务
3. `BLOCKED`
   - 线程因缺信息、缺依赖、缺权限或缺决策而受阻
4. `NEEDS_COLLAB`
   - 线程判断需要其他线程协作
5. `WAITING`
   - 线程已提交请求或阶段输出，等待外部回应
6. `HANDOFF_READY`
   - 线程已准备好 handoff，适合继续接力
7. `DONE`
   - 线程当前任务已完成
8. `DORMANT`
   - 线程暂时待命或长时间无更新

## 5. 推荐阶段值
为了提高运行中的可解释性，线程卡推荐补充 `phase`：
1. `DISCOVERING`
2. `BOUNDING`
3. `IMPLEMENTING`
4. `VERIFYING`
5. `HANDING_OFF`
6. `CLOSING`

## 6. 协作请求类型
当前推荐固定使用以下请求类型：
1. `CONSULT`
2. `IMPLEMENT`
3. `VERIFY`
4. `REVIEW`
5. `INTEGRATE`
6. `UNBLOCK`

## 7. 事件驱动更新规则
线程不需要频繁写状态，只在以下事件发生时更新：
1. 第一次接手任务
2. 正式进入执行
3. 发现阻塞
4. 需要协作
5. handoff 准备完成
6. 当前任务完成
7. 进入待命

这类更新属于“状态切换事件”，应优先于过程流水记录。

## 8. 反馈信号
线程状态卡和协作请求卡推荐显式表达：
1. `confidence`
2. `deviation_flag`
3. `decision_needed`
4. `recommended_next_step`
5. `acceptance_signal`

这些字段的目标是让管理线程能更快判断：
1. 线程是不是可能偏航
2. 当前是否需要介入或确认
3. 当前协作请求是否真的可接

## 9. 过期与待命规则
1. 如果线程在当前任务周期内长期无更新，可由管理线程将其状态调整为 `DORMANT`
2. 如果协作请求在合理窗口内无人处理，可将请求状态调整为 `STALE`
3. `DORMANT` 与 `STALE` 都属于运行态判断，不应直接写入长期知识层
4. 如果线程恢复推进，应从 `DORMANT` 切回实际状态，而不是新建平行线程记录

## 10. 自动写入边界
以下内容允许线程自动写入：
1. 自己的线程状态卡
2. 自己发起的协作请求
3. handoff 草稿与交接材料
4. deliverables 中的线程草稿

以下内容仍建议保留人工确认：
1. `10-services`
2. `20-chains`
3. `40-evidence`
4. `50-decisions`
5. `90-archive`

说明：
1. 动态运行态信息是低风险、可覆盖、可过期的
2. 长期知识层是高价值、低容错的

## 11. 写入责任
1. 执行线程
   - 只更新自己的线程状态卡
   - 只创建或更新自己发起的协作请求
2. 管理线程
   - 维护 `thread-board.md`
   - 汇总全局协作状态
   - 决定是否派发、纠偏、收口、归档

## 12. 非目标
当前阶段不做以下事情：
1. 线程之间直接通信
2. 自动派发执行线程
3. 自动决定协作者
4. 自动归档进入长期知识层

当前阶段优先目标是：
1. 让线程存在可见
2. 让线程状态可见
3. 让协作意图可见
4. 让偏航、低信心和决策需求可见

## 13. 一句话原则
线程协作运行层服务的是“运行中的协作可见性”，不是“正式长期知识沉淀”。
