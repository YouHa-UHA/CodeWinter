# CodeWinter Release Governance v1 发布治理

本文定义 `CodeWinter` 本体发布的成熟度治理规则。

它回答的是：
1. 一版 `CodeWinter` 什么时候只是草案。
2. 什么时候可以进入试点接入。
3. 什么时候可以作为默认推荐基线。
4. 不同成熟度下，兼容承诺、迁移要求和文档义务有多强。

## 1. 核心原则
1. 发布通道表达的是成熟度，不是版本大小。
2. 发布治理与版本模型配合使用，但不替代版本判断。
3. 越高成熟度的发布通道，越强的兼容承诺，越严格的发布说明义务。
4. 发布通道要帮助管理者判断“能不能接项目”，而不是只给一个标签。

## 2. 发布通道
`CodeWinter` 当前固定三档发布通道：
1. `draft`
2. `candidate`
3. `stable`

## 3. `draft`
适用场景：
1. 本体还在快速演进。
2. 核心结构仍可能调整。
3. 协议与术语还在收敛中。

默认承诺：
1. 只提供最低限度的结构一致性承诺。
2. 不承诺大范围接入稳定性。
3. 可以继续进行结构级优化。

推荐使用范围：
1. 本体设计期。
2. 小范围内部验证。
3. 由你主导的实验性接入。

文档要求：
1. 必须有 `release-manifest.md`。
2. 结构变化、兼容边界变化时，建议发布 release notes。
3. 如果涉及实例迁移风险，必须补 migration guidance。

## 4. `candidate`
适用场景：
1. 核心结构已经基本收敛。
2. 准备给真实项目做试点接入。
3. 仍允许修正，但不应频繁改主边界。

默认承诺：
1. 对当前声明的结构轴和运行轴提供明确支持边界。
2. 允许滚动升级和兼容期治理。
3. 不应再随意修改核心术语和必需工件。

推荐使用范围：
1. 试点项目。
2. 内部受控接入。
3. 准备晋级 `stable` 前的真实运行验证。

文档要求：
1. 必须有 `release-manifest.md`。
2. 必须有 release notes。
3. 必须声明迁移影响、兼容期建议和回滚思路。

## 5. `stable`
适用场景：
1. 可作为默认推荐接入基线。
2. 兼容承诺应严格执行。
3. 不应再出现未声明的结构级破坏性变化。

默认承诺：
1. 对当前声明的最小支持结构轴和运行轴提供清晰兼容边界。
2. 结构变化和运行协议变化必须显式进入迁移流程。
3. 发布说明必须足够支持正在运行的项目做迁移决策。

推荐使用范围：
1. 默认推荐的新项目接入。
2. 正在运行项目的常规跟进升级。
3. 需要较强稳定性的协作系统基线。

文档要求：
1. 必须有 `release-manifest.md`。
2. 必须有 release notes。
3. 必须包含迁移影响、兼容期建议、冻结建议和回滚说明。

## 6. 晋级规则
### 6.1 `draft -> candidate`
建议至少满足：
1. 版本模型已经稳定。
2. 发布清单已经稳定。
3. Bootstrap、Upgrade & Migration、线程协作运行层已形成闭环。
4. 不存在明显的 P0 级结构缺口。
5. 至少完成一次完整的本体自检或 dry run。

### 6.2 `candidate -> stable`
建议至少满足：
1. 已完成至少一次真实接入验证或等价强度验证。
2. 已验证至少一次升级或迁移路径。
3. 关键治理边界已经稳定：
   - 结构轴
   - 运行轴
   - 归档边界
   - 管理线程职责边界
4. release notes 已能稳定支持迁移决策。
5. 没有未收口的高风险协议争议。

## 7. 降级与纠偏规则
如果 `candidate` 或 `stable` 发布后发现严重问题：
1. 优先通过后续 patch 或 minor release 修正。
2. 不建议静默改写旧发布的含义。
3. 如确有必要，应显式发布风险说明，并重新声明推荐采用通道。

说明：
1. 发布通道可以调整。
2. 但不应让旧版本“悄悄换了定义”。

## 8. 发布通道与变更预算
### 8.1 `draft`
允许：
1. 结构级重构。
2. 术语级收敛。
3. 流程级重组。

### 8.2 `candidate`
允许：
1. patch 修正。
2. 低风险 minor 增强。
3. 明确说明下的兼容扩展。

不建议：
1. 未经说明的结构级破坏性变化。
2. 无迁移计划的运行协议变化。

### 8.3 `stable`
允许：
1. patch 修正。
2. 兼容性明确的 minor 增强。
3. 带正式迁移计划的 major 演进。

不允许：
1. 未声明的 breaking change。
2. 绕过迁移协议直接改核心边界。

## 9. 发布通道与采用建议
1. `draft`
   - 默认不作为大范围项目接入基线。
2. `candidate`
   - 适合作为试点接入基线。
3. `stable`
   - 适合作为默认推荐接入基线。

## 10. 与 release notes 的关系
发布治理回答的是：
1. 这版处于什么成熟阶段。
2. 兼容承诺应有多强。
3. 是否适合被大范围采用。

release notes 回答的是：
1. 这次具体改了什么。
2. 影响谁。
3. 该怎么迁移。

二者必须同时存在，但职责不同。

## 11. 一句话原则
`release_channel` 决定“这版能不能放心推广”，release notes 决定“这版到底该怎么升级”。
