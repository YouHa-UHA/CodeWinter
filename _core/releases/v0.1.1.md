# CodeWinter v0.1.x Harness Upgrade

## 发布摘要
1. `release_version`: `v0.1.1`
2. `release_channel`: `draft`
3. `release_theme`: `CodeWinter v0.1.x Harness Upgrade`
4. `release_codename`: `Carrot on a Stick 胡萝卜钓竿`
5. 主影响标签：`MIGRATION_RECOMMENDED`
6. 一句话摘要：将 `CodeWinter` 从“规则优先”进一步升级为“引导优先”，正式启用发布记录目录，并在不提升 `schema-v1` / `runtime-v1` 的前提下强化 Harness 能力。

## 当前发布留档
1. 项目介绍：`./CodeWinter/_core/releases/v0.1.1.project-introduction.md`
2. 使用说明：`./CodeWinter/_core/releases/v0.1.1.usage-guide.md`

说明：
1. 这两份文件代表当前 `v0.1.1` 发布线下的最新人类可读口径。
2. 如果只是说明优化、措辞修正或非协议级补充，可以原地更新，不必强制发布新版本。

## 为什么发布
这次发布主要解决两个问题：
1. 当前 `CodeWinter` 已具备分层、运行态协作、实例迁移和版本治理，但对线程行为的默认引导仍偏弱。
2. 版本治理模型已经存在，却还没有真正启用连续发布记录目录。

因此，本次发布一方面把 `Harness Engineering` 正式纳入系统核心思想，另一方面正式启用 `_core/releases/` 记录本体发布说明。

## 变更摘要
### `_core`
1. 将 `Harness Engineering` 写入设计原则与术语表。
2. 为发布清单补充 `release_theme`、`release_codename` 和 `release_notes_path`。
3. 正式启用 `_core/releases/` 作为发布说明目录。

### 控制面
1. `manager-brief.md` 增加当前控制目标、编排策略、管理介入条件和收口条件。
2. `thread-board.md` 增强为能暴露阶段、信心、推荐下一步和管理关注项的总览页。
3. `instance-manifest.md` 与当前本体发布元数据对齐。

### 线程协作运行层
1. 线程状态卡新增 `phase`、`confidence`、`deviation_flag`、`decision_needed`、`recommended_next_step` 等反馈字段。
2. 协作请求卡新增 `why_now`、`recommended_receiver_profile`、`acceptance_signal` 等引导字段。
3. 运行层规则从“状态可见”升级为“状态、偏航和决策需求可见”。

### 协作协议与提示词
1. 执行协议加入默认执行回路。
2. `Green / Yellow / Red` 从风险标签升级为控制门。
3. `thread-start`、`thread-resume`、`thread-runtime-update`、`thread-handoff`、`collab-request` 全部注入 Harness 引导。
4. `Bootstrap` 阶段新增控制目标与编排策略的初始化要求。

### 收口与删除
1. 删除独立的 `risk-levels.md`。
2. 将其核心逻辑并入 `ai-collaboration-protocol.md`，避免风险模型继续停留在静态说明层。

## 兼容性影响
1. `instance_schema_target`: `schema-v1`
2. `instance_schema_min_supported`: `schema-v1`
3. `runtime_coordination_target`: `runtime-v1`
4. `runtime_coordination_min_supported`: `runtime-v1`
5. 本次发布不提升结构轴和运行轴。

受影响实例：
1. 所有已在使用 `CodeWinter` 多线程协作的项目实例都可采用本次升级。
2. 旧实例如果仍使用旧版线程卡、协作请求卡或独立风险说明页，建议跟进。

## 管理动作
### 新接入项目
1. 直接使用 `v0.1.1` 作为本体基线。
2. Bootstrap 时同时初始化控制目标、编排策略和管理介入条件。

### 正在运行的实例
1. 优先升级 `core-owned` 内容和管理工具箱模板。
2. 线程卡和协作请求卡可以滚动补齐新字段，不要求一次性全量重写。
3. 如实例中仍保留 `risk-levels.md` 依赖，应将风险判断迁移到 `ai-collaboration-protocol.md` 的控制门逻辑。

### 活跃线程
1. 已在执行中的线程可先按旧上下文收尾。
2. 下一次 `Start` / `Resume` 时，应切换到新的默认执行回路与反馈字段。

## 迁移要求
1. 是否需要实例迁移：不需要强制结构迁移。
2. 是否需要兼容期：可按滚动升级处理。
3. 是否建议冻结新任务派发：不需要。
4. 关键迁移动作：
   - 更新发布清单与实例清单字段
   - 更新管理简报模板
   - 更新线程卡和协作请求卡模板
   - 用协议内控制门替代独立风险说明页

## 废弃项与风险
1. 废弃做法：把 `Green / Yellow / Red` 当成只读标签，而不让它影响默认动作。
2. 废弃文件：独立 `risk-levels.md`。
3. 当前风险：
   - 旧实例中的活跃线程可能仍按旧模板工作
   - 如果只升级部分模板，可能出现新旧提示词并存的短期不一致

## 回滚提醒
1. 如果升级后发现模板措辞导致线程普遍误判，应优先回滚 `core-owned` 模板与协议文件。
2. 回滚时优先保护 `instance-owned` 内容，包括任务包、正式输出、长期知识层和运行中的线程记录。
3. 不要为了回滚本体模板而覆盖实例自己的动态控制面与历史知识。
