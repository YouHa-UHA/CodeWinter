# CodeWinter 设计原则

本文定义 `CodeWinter` 的底层设计思想。

## 1. Progressive Disclosure（渐进式披露）
线程启动时，只先读取最小必要上下文。

默认顺序：
1. Starter
2. 当前控制面简报
3. 协作规则
4. 如果有任务包，再读任务包
5. 最后才读服务、链路、证据、决策

这样做的原因：
1. 降低线程启动成本。
2. 减少上下文污染。
3. 让执行线程保持边界清晰。

## 2. Bounded Context Workers（边界化执行线程）
执行线程一次只负责一个清晰切片：
1. 一个模块
2. 一个服务
3. 一条链路
4. 一个交付物
5. 一个验证问题

这样做的原因：
1. 上下文更短更准。
2. 交接更稳定。
3. 多线程扩展更安全。

## 3. Externalized State（状态外部化）
重要项目知识必须写入文件，而不是只留在聊天上下文里。

这样做的原因：
1. 线程可以替换。
2. 工具可以切换。
3. 中断后可以恢复。

## 4. Role Lease, Not Permanent Thread（角色租约，而非永久线程）
管理线程是一个租约角色，不是永久会话身份。

这意味着：
1. 当前可以由某个线程持有管理租约。
2. 之后可以由另一个线程接管同一角色。
3. 连续性由控制面定义，而不是由单个聊天窗口保证。

这样做的原因：
1. 避免系统依赖单线程存活。
2. 让系统天然可迁移到不同客户端。
3. 连续性来自显式状态，而不是隐式记忆。

## 5. Artifact-First Coordination（以工件为先的协作）
线程之间应优先通过可落盘工件协作：
1. 控制面文件
2. 任务包
3. handoff
4. deliverables
5. evidence 和 decisions

这样做的原因：
1. 工件可检查。
2. 工件可回放。
3. 工件能跨工具继续使用。

## 6. Harness Engineering（支架工程）
`CodeWinter` 不只约束线程，更通过默认路径、阶段门、状态机、工件模板、反馈回路和纠偏机制，引导线程稳定地产生高质量协作行为。

默认思路：
1. 让管理线程保持清晰的控制目标和编排策略。
2. 让执行线程按默认执行回路推进，而不是靠临场发挥。
3. 让风险等级变成真正的控制门，而不是静态标签。
4. 让运行态工件暴露信心、偏航和决策需求。

这样做的原因：
1. 好的系统应让线程更容易做对，而不是只在做错后补救。
2. 多线程协作的稳定性更多来自 Harness 质量，而不只来自模型本身。
3. 用户不应长期承担高强度人工纠偏成本。

## 7. Evidence Before Memory（先证据，后记忆）
长期记忆必须从已采纳、已证实的信息中提炼，而不是直接把输入和聊天结果当真相。

默认规则：
1. 原始输入不等于长期记忆。
2. 对话输出默认也不等于长期记忆。
3. 只有已采纳、已证实、可复用的事实才进入长期层。

这样做的原因：
1. 防止记忆漂移。
2. 防止把试错写成系统真相。
3. 让知识库保持小而可信。

## 8. Human Approval Gates（人工确认闸口）
高影响事项必须经过人工确认。

典型场景：
1. 表结构变化
2. 外部接口契约变化
3. 权限和安全模型变化
4. 跨服务行为口径变化
5. 高价值决策的正式归档

这样做的原因：
1. AI 应该加速决策，而不是偷偷替代决策。
2. Red 风险任务里，治理质量比速度更重要。

## 9. Structured Uncertainty（结构化不确定性）
未知信息必须显式写成 `to confirm`，而不是靠猜测补齐。

这样做的原因：
1. 隐性不确定性比显性不确定性更危险。
2. 团队只有在未知项被暴露出来时，才能做出好判断。

## 10. Verification Loops（验证闭环）
线程处理任务时，应优先形成闭环：
1. 理解
2. 方案
3. 改动
4. 验证
5. 归档

这样做的原因：
1. 让 AI 工作方式更像工程协作。
2. 减少“看起来完成了，实际上没闭环”的浅层完成。

## 11. Tool Portability（工具可迁移）
`CodeWinter` 不能依赖某一个 AI 客户端。

基本要求：
1. 使用相对路径。
2. 使用普通 Markdown 和普通文本提示词。
3. 核心协议中不写客户端私有指令。
4. 默认假设当前线程可能几乎没有历史记忆。

这样做的原因：
1. 同一个项目可以在 Codex、Cursor、Claude 等工具之间切换。
2. 防止控制面被单一产品锁死。
