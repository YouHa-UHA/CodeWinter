# CodeWinter Upgrade & Migration v1 升级与迁移协议

本文定义 `CodeWinter` 在已运行项目中的升级方式。

核心结论只有一句话：

**`CodeWinter` 的升级不是覆盖文件夹，而是实例迁移。**

也就是说，未来每次升级都应被视为：
1. 上游 `CodeWinter release` 发布
2. 某个项目实例执行 `instance migration`
3. 经过兼容期后完成切换

## 1. 为什么需要升级协议
当 `CodeWinter` 只是一组静态文档时，直接替换文件问题不大。  
但当 `CodeWinter` 已经承担：
1. 控制面
2. 线程运行态
3. 任务输入
4. 正式输出
5. 长期知识

它就不再是普通模板，而是一个正在运行的项目实例。

此时如果仍然按“直接覆盖文件”升级，会有三类典型风险：
1. 覆盖项目实例自己的动态状态
2. 让活跃线程与新协议断裂
3. 混淆平台资产与实例数据的边界

## 2. 核心概念
### 2.1 `CodeWinter release`
指 `CodeWinter` 上游平台发布的一版规范。

它通常包含：
1. 核心协议
2. 通用模板
3. 通用目录说明
4. 通用工具箱

当前发布基线应以 `./CodeWinter/_core/release-manifest.md` 为准。

### 2.2 `CodeWinter instance`
指某个具体项目中正在运行的一份 `CodeWinter` 实例。

它通常包含：
1. 当前控制面
2. 当前线程运行态
3. 当前任务输入与正式输出
4. 当前长期知识

### 2.3 `instance migration`
指将某个项目实例从旧版 `CodeWinter` 升级到新版 `CodeWinter` 的过程。

### 2.4 `compatibility window`
指实例升级后的一段兼容期。

在兼容期内：
1. 新派发线程按新协议运行
2. 已在执行中的旧线程允许按旧协议收尾
3. 管理线程负责桥接新旧协议

### 2.5 版本判断原则
实例升级不能只看 `release_version`。

必须同时比较：
1. 发布版本
2. 实例结构版本
3. 运行协作版本

具体规则以 `./CodeWinter/_core/versioning-model-v1.md` 为准。

## 3. 所有权边界
升级前必须先区分三类内容。

### 3.1 `core-owned`
指可由上游版本安全替换或升级的内容。

典型包括：
1. `_core/*`
2. 通用规则文档
3. 通用工具箱模板
4. 通用目录说明

特点：
1. 属于平台规范
2. 原则上可由升级过程自动更新

### 3.2 `instance-owned`
指某个项目实例自己的运行数据和知识资产。

典型包括：
1. `00-control-plane/manager-brief.md`
2. `00-control-plane/active-queue.md`
3. `00-control-plane/thread-board.md`
4. `00-control-plane/threads/*`
5. `00-control-plane/collab-requests/*`
6. `04-task-packets/*`
7. `05-deliverables/*`
8. `10-services/*`
9. `20-chains/*`
10. `30-handoffs/*`
11. `40-evidence/*`
12. `50-decisions/*`
13. `90-archive/*`

特点：
1. 属于项目实例
2. 不应被升级流程直接覆盖

### 3.3 `managed-with-migration`
指可以升级，但必须通过迁移处理，而不是直接覆盖的内容。

典型包括：
1. 控制面结构升级
2. 模板字段升级
3. 状态机字段升级
4. 运行态协作层结构升级

特点：
1. 需要迁移计划
2. 需要兼容期或版本桥接

## 4. 升级分级
### 4.1 Patch Upgrade
适用于：
1. 文案优化
2. 模板细化
3. 新增非破坏性字段
4. 不改变旧结构的规则增强

特点：
1. 通常可原地升级
2. 一般不需要冻结线程派发
3. 风险最低

### 4.2 Minor Upgrade
适用于：
1. 新增协作能力层
2. 新增目录但不破坏旧目录
3. 新增状态机或请求卡机制
4. 旧流程仍可兼容运行

特点：
1. 适合滚动升级
2. 管理线程先升级
3. 新任务走新协议
4. 旧线程允许兼容收尾

### 4.3 Major Upgrade
适用于：
1. 调整核心目录边界
2. 调整知识层映射
3. 调整线程生命周期模型
4. 调整核心术语与主协议
5. 调整长期知识结构

特点：
1. 必须有正式迁移计划
2. 可能需要短暂冻结新任务派发
3. 必须明确回滚边界

## 5. 升级总流程
### 第一步：识别版本差异
先回答三个问题：
1. 当前实例运行在哪个版本
2. 目标升级版本是什么
3. 本次升级属于 Patch / Minor / Major 哪一类

优先读取：
1. `./CodeWinter/00-control-plane/instance-manifest.md`
2. `./CodeWinter/_core/versioning-model-v1.md`
3. `./CodeWinter/_core/release-manifest.md`

### 第二步：生成升级计划
在正式升级前，应先在 `./CodeWinter/00-control-plane/upgrade-plan.md` 输出一份升级计划，至少说明：
1. 本次升级类型
2. 影响范围
3. 受影响目录
4. 哪些内容可自动更新
5. 哪些内容必须人工确认
6. 是否需要兼容期
7. 是否需要冻结新任务派发
8. 风险与回滚思路

### 第三步：保护实例动态状态
升级前至少应保护以下内容：
1. `manager-brief.md`
2. `active-queue.md`
3. `thread-board.md`
4. 线程状态卡
5. 协作请求卡

说明：
1. 这里的“保护”可以是备份、快照、归档副本，或其他不会影响实例继续恢复的方式

### 第四步：应用核心层升级
此阶段只更新 `core-owned` 内容。

### 第五步：执行实例迁移
如果本次升级涉及：
1. 新目录
2. 新模板字段
3. 新状态机
4. 新协作层结构

则在此阶段补齐或迁移这些实例结构。

注意：
1. 迁移实例结构不等于覆盖实例数据

### 第六步：进入兼容期
兼容期内遵循以下规则：
1. 新派发线程按新协议工作
2. 老线程允许按旧协议完成当前任务
3. 老线程如要进入下一阶段，应用新协议 `Resume`
4. 管理线程负责桥接新旧线程

### 第七步：升级完成与收口
升级完成后，应至少完成：
1. 版本状态更新
2. 兼容期状态确认
3. 遗留问题登记
4. 升级结果回顾

并同步更新：
1. `./CodeWinter/00-control-plane/instance-manifest.md`
2. `./CodeWinter/00-control-plane/upgrade-log.md`

## 6. 活跃线程处理规则
### 6.1 不强行重写活跃线程
已在执行中的线程，不建议在升级瞬间强制切到新协议。

更稳妥的方式是：
1. 允许其完成当前任务或当前阶段
2. 下一阶段再按新协议 `Resume`

### 6.2 新线程总是走新版
只要实例升级已完成，新派发线程一律读新版控制面和新版协议。

### 6.3 老线程长期继续时，需要协议重接
如果旧线程仍要继续长期参与协作，应：
1. 重新读 Starter
2. 重新读管理简报
3. 重新读升级说明
4. 更新自己的线程状态卡

## 7. 自动与人工的边界
### 7.1 可半自动升级的内容
1. `_core` 层
2. 控制面模板
3. 线程协作运行层结构
4. 新目录初始化
5. 版本清单
6. 状态机字段扩展

### 7.2 继续人工确认的内容
1. `10-services`
2. `20-chains`
3. `40-evidence`
4. `50-decisions`
5. `90-archive`

原因：
1. 动态协作层是低风险、可覆盖、可过期的
2. 长期知识层是高价值、低容错的

## 8. 冻结与兼容策略
### 8.1 什么时候不需要冻结
一般来说，以下情况可不冻结：
1. Patch Upgrade
2. 低风险 Minor Upgrade
3. 不影响活跃线程主路径的增强项

### 8.2 什么时候建议冻结新任务派发
以下情况建议短暂冻结新任务派发：
1. Major Upgrade
2. 控制面主协议变化
3. 目录边界变化
4. 线程生命周期变化

说明：
1. “冻结”通常是指暂停派发新任务，不一定是让所有活跃线程全部停机

## 9. 回滚原则
如果升级过程中出现明显问题，应至少能够回答：
1. 回滚到哪个版本
2. 哪些核心文件可以恢复
3. 哪些实例数据不能回滚覆盖
4. 哪些线程需要重新接手或重新登记状态

回滚的核心原则是：
1. 优先保护实例数据
2. 避免用旧 core 覆盖新实例数据
3. 回滚后要重新建立控制面一致性

## 10. 最小落地元信息
为了让升级协议真正可执行，`CodeWinter` 实例默认应落盘以下升级元信息：
1. `./CodeWinter/00-control-plane/instance-manifest.md`
   - 记录当前实例版本、结构版本、运行状态与兼容期状态
2. `./CodeWinter/00-control-plane/upgrade-plan.md`
   - 记录当前升级前计划；无升级时保持最小占位
3. `./CodeWinter/00-control-plane/upgrade-log.md`
   - 记录已执行升级的结果与遗留事项

这样做的原因是：
1. 协议必须有稳定落点，管理线程才能做滚动升级和回滚判断
2. 实例迁移状态必须外部化，不能只留在某次对话里

## 11. 非目标
当前升级协议不追求：
1. 自动选择协作者
2. 自动升级所有活跃线程上下文
3. 自动写入长期知识层
4. 用一个统一脚本强行覆盖所有项目实例

## 12. 一句话原则
已运行项目中的 `CodeWinter` 升级，应被视为“控制面协议迁移”，而不是“替换模板文件”。
