# CodeWinter 工具可迁移约定

`CodeWinter` 应当尽量在不同 AI coding 工具之间无痛运行。

## 1. 最低可迁移能力基线
控制面只默认依赖以下能力：
1. 读文件
2. 写文件
3. 搜代码
4. 跑命令
5. 进行普通对话

任何超出这些能力的特性，都应该被视为加速器，而不是系统硬依赖。

## 2. 提示词设计规则
所有共享提示词都应满足：
1. 使用纯文本或 Markdown
2. 使用相对路径
3. 不依赖客户端私有指令
4. 明确说明输出要求和停止条件

避免：
1. 产品专属 UI 假设
2. 隐式系统提示词依赖
3. 机器专属绝对路径
4. 只有某个客户端才能理解的操作说明

## 3. 路径规则
优先使用项目根目录相对路径，例如：
1. `./CodeWinter/read.md`
2. `./CodeWinter/10-services/<service>/service-manifest.md`
3. `./apps/web/src`

不要依赖机器专属绝对路径。

## 4. 记忆规则
默认假设不同 AI 工具在以下方面都可能不同：
1. 上下文窗口大小
2. 历史持久化行为
3. 搜索能力
4. 工具可用性

因此：
1. 耐久状态必须落入文件。
2. 大上下文要被摘要成可复用工件。
3. 线程应当总能从文件系统恢复。

## 5. 恢复约定
如果线程丢失，或工具发生切换：
1. 新线程先读 `./CodeWinter/read.md`
2. 再读 `./CodeWinter/00-control-plane/manager-brief.md`
3. 再读 `./CodeWinter/01-thread-rules/ai-collaboration-protocol.md`
4. 然后根据任务包、服务卡、链路卡、证据层、决策层继续恢复

这是最小连续性约定。

## 6. 可选加速器
以下能力如果存在可以利用，但不应成为 `CodeWinter` 的前提：
1. 子代理 / sub-agents
2. 原生 code review UI
3. 可视化 diff 工具
4. 客户端自带长期记忆
5. MCP、插件、连接器

`CodeWinter` 应该能从这些能力中获益，但不应该被它们绑定。
